## Warnings

### WR-01: Local-time `todayIso()` is inconsistent with UTC-based `daysUntil()` — can produce a one-day error at midnight

**File:** `src/lib/legal/calculate-deadlines.ts:69-75`

**Issue:** `todayIso()` builds a date string from local wall-clock time (`getFullYear()`, `getMonth()`, `getDate()`). `daysUntil()` in the same file (line 57-67) computes "today UTC" via `Date.UTC(today.getFullYear(), today.getMonth(), today.getDate())` — also local time but then treated as UTC. Both functions agree on the date string when the server timezone matches UTC. However, `computedAt` in the result is produced by `todayIso()` (local time) while `daysRemaining` is produced by `daysUntil()` using UTC arithmetic on the target date. When the server runs in a timezone behind UTC (e.g., UTC-8) the "today" used for `daysRemaining` can be one calendar day behind the "today" written to `computedAt`, producing a one-day discrepancy in displayed remaining days. The same issue exists in `calculate-prescription.ts:59-68`.

**Impact:** Legal deadlines are the core output of this service. A one-day off-by-one in `daysRemaining` near a deadline could cause a taxpayer to believe they have an extra day.

**Fix:**
```typescript
// Replace todayIso() with a UTC-anchored version:
function todayIso(): string {
  const now = new Date();
  const yyyy = now.getUTCFullYear();
  const mm = String(now.getUTCMonth() + 1).padStart(2, "0");
  const dd = String(now.getUTCDate()).padStart(2, "0");
  return `${yyyy}-${mm}-${dd}`;
}

// And in daysUntil(), align to UTC today:
function daysUntil(iso: string): number {
  const now = new Date();
  const todayUTC = Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate());
  const target = new Date(`${iso}T00:00:00.000Z`).getTime();
  return Math.round((target - todayUTC) / (1000 * 60 * 60 * 24));
}
```

Apply the same fix to the duplicate `daysUntil()` in `src/lib/legal/calculate-prescription.ts:59-68`.

### WR-02: Signatory authorization check uses `includes()` — allows false positives via substring match

**File:** `src/lib/legal/check-waiver-validity.ts:114-116`

**Issue:** The authorized signatory check uses:
```typescript
signatoryNormalized === role || signatoryNormalized.includes(role)
```
The `includes(role)` half means any input string that contains an authorized role title as a substring passes. For example, a signatory of `"Revenue District Officer for the Regional Director"` would match `"regional director"` because that substring appears in it. An unauthorized officer claiming to act on behalf of an authorized one — or a typo that happens to include the role name — would be passed as valid.

**Impact:** This is a legal correctness bug. Its purpose is to detect defective waivers; false negatives (passing an unauthorized signatory as authorized) could cause the AI to advise a taxpayer that their waiver is valid when it is not.

**Fix:** Flip the direction. Check whether the authorized role is contained in the input, not the other way around, but do so more precisely by normalizing to exact match or anchoring the substring check:
```typescript
// Prefer exact match; only fall back to substring if the input might
// contain additional context like "duly authorized" or full name + title.
// The safest approach is exact-match only and let the caller normalize the input:
const isAuthorized = AUTHORIZED_SIGNATORIES.some(
  (role) => signatoryNormalized === role
);
```
If the AI may pass verbose role strings (e.g., "Regional Director, Revenue Region No. 8"), consider `signatoryNormalized.includes(role)` rather than `role.includes(signatoryNormalized)` — which is the current (wrong) direction. The current code has the substring operands reversed relative to intent.

### WR-03: Missing error handling in `assessComplexity` escalation tool — email failure silently bubbles up as tool error

**File:** `src/lib/ai/tools.ts:276-287`

**Issue:** The escalation `execute` block awaits both `createEscalation()` and `sendEscalationNotificationEmail()` without any try/catch. If the DB write succeeds but the email send fails (transient Resend error, network timeout), the tool throws an unhandled rejection to the Vercel AI SDK. This causes the entire chat turn to error out, even though the escalation record was already persisted. The taxpayer sees a chat failure rather than a success acknowledgment.

**Fix:**
```typescript
execute: async (input) => {
  if (!input.isComplex) {
    return { escalated: false, message: "Case does not require escalation." };
  }

  const reasons: string[] = [];
  if (input.largeTaxAmount) reasons.push("Large tax amount (>PHP 1,000,000)");
  if (input.multipleLOADefects) reasons.push("Multiple LOA validity defects");
  if (input.sdtInvolvement) reasons.push("Subpoena Duces Tecum involvement");
  if (input.multipleTaxPeriods) reasons.push("Multiple tax periods under examination");
  if (input.fraudAllegations) reasons.push("Fraud allegations by BIR");
  if (input.conflictingLegalGrounds) reasons.push("Conflicting legal grounds");

  await createEscalation({
    sessionId,
    summary: input.summary,
    complexityReasons: reasons,
    severity: input.severity,
  });

  // Email is best-effort — don't let a transient failure abort the chat turn
  try {
    await sendEscalationNotificationEmail({
      summary: input.summary,
      reasons,
      severity: input.severity,
      sessionEmail,
    });
  } catch (emailErr) {
    // Log for ops visibility but don't re-throw; escalation record is already saved
    console.error("[escalation] email notification failed:", emailErr);
  }

  return { escalated: true, message: "Case has been flagged for CPA review." };
},
```

**File:** `src/lib/legal/calculate-deadlines.ts:57-67` and `src/lib/legal/calculate-prescription.ts:59-68`

**Issue:** The `daysUntil()` helper function is copy-pasted identically into both files. This means any fix (such as WR-01 above) must be applied twice, and the two can drift out of sync.

**Fix:** Extract to `src/lib/legal/parse-flexible-date.ts` or a new `src/lib/legal/date-utils.ts` shared module and import from both callers:
```typescript
// date-utils.ts
```
